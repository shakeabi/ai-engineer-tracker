W08

Flagship: unified observability platform

The centerpiece deliverable so far — not a side feature

0/0 ▾

By now you have three systems (PR review agent, research agent, fine-tuned model) each emitting traces, costs, and latency data using the same instrumentation approach. This week, stop adding new AI capability and instead build the thing that ties it all together: a single observability platform that watches all three systems at once. This is the artifact that makes you different from every other AI engineering candidate — most of them have never built monitoring for anything, let alone for three heterogeneous LLM systems simultaneously.

Learn

  • What to trace across heterogeneous systems: latency, tokens in/out, cost, model version, cache hits — and how to normalize these across system types (agent vs fine-tune vs RAG)
  • Prompt caching and how it cuts cost on repeat calls
  • Rate limit handling across multiple concurrent users
  • Semantic caching: avoid redundant LLM calls for similar inputs

Build

  • Unify OpenTelemetry traces from Weeks 5, 6, and 7 into a single collector/backend
  • One Grafana dashboard showing all three systems: cost, latency, cache hit rate, error rate
  • Semantic caching with Redis + embedding similarity threshold, applied across systems where relevant
  • Alerts on cost anomalies and latency p95 spikes, system-agnostic

W15

vLLM fundamentals + model serving

Understand what's actually happening between "model" and "API endpoint"

0/0 ▾

Learn

  • Why naive Hugging Face generate() doesn't scale — the throughput problem
  • Continuous batching: how vLLM serves many concurrent requests efficiently
  • PagedAttention: the memory management technique behind vLLM's efficiency
  • Throughput vs latency tradeoffs in LLM serving, and how to tune for each

Build

  • Serve your Week 7 fine-tuned model with vLLM instead of raw transformers
  • Benchmark throughput (tokens/sec) and latency (p50/p95) before and after switching to vLLM
  • Load test with concurrent requests to see where the naive setup breaks and vLLM doesn't
  • Document GPU memory usage under load

W16

Kubernetes deployment for LLM workloads

Your differentiator: deploying AI the way production infra actually works

0/0 ▾

Learn

  • GPU scheduling basics in Kubernetes — node selectors, resource limits, GPU operators
  • Autoscaling LLM workloads — why standard HPA metrics (CPU) don't work well, and what does (queue depth, latency-based)
  • Health checks and readiness probes for slow-starting model containers
  • Cost implications of GPU node pools — why this differs from scaling a stateless API

Build

  • Containerize your Week 15 vLLM service
  • Deploy to a k8s cluster (local via kind/minikube, or a managed cluster if you have cloud credits)
  • Set up autoscaling based on a meaningful metric (queue depth or request latency, not just CPU)
  • Wire this deployment into your Week 8 observability platform — treat it as a 5th monitored system

A Kubernetes-deployed, autoscaling LLM service — the capstone of Month 4 and the whole 16-week roadmap.

Expected features

// what to skip

✗ Skip

Jupyter notebooks as deliverables

Notebooks are for exploration. Real engineers ship APIs and services. Every project should end with something deployed and callable, not a .ipynb file.

✗ Skip

Certifications as portfolio items

DeepLearning.AI courses are fine for learning, but don't lead with them on a resume. They signal that you watched something, not that you built something.

✗ Skip

Projects with no evals

If you can't measure it, you didn't engineer it. Every project needs at least one quantitative metric — recall@k, cost/request, latency p95, accuracy delta.

✗ Skip

"I wrapped GPT-4 in FastAPI"

The market is drowning in these. Chunking strategies, hybrid retrieval, eval harnesses, cost controls — that's what distinguishes engineering from duct tape.
